# Context Optimization and Automated Re-hydration Design

## Summary
This project implements a "Hot/Cold" context management system designed to maintain high performance in long-running agent sessions without exceeding token limits. The core strategy involves aggressively pruning active `SKILL.md` instructions down to a "Hot" core of mandatory constraints while offloading detailed "Cold" domain knowledge into secondary `REFERENCE.md` files. These reference files are only loaded by the agent when a specific task requires that domain expertise, significantly reducing the baseline prompt overhead and freeing up context for complex reasoning.

To ensure continuity across mandatory context resets, the system introduces automated session journaling and re-hydration. A background monitoring hook captures critical milestones—such as file modifications, architectural decisions, and phase completions—into a machine-readable journal. When a session is cleared (either manually via `/clear` or automatically due to context pressure), a startup hook identifies the most recent journal entry and re-seeds the new session with its essential state. This allows the agent to resume work with a full understanding of the project's intent and current progress, effectively eliminating the "amnesia" that typically follows a context wipe.

## 🔴 Definition of Done
- All `SKILL.md` files in the `plugins/` directory are refactored to be under the 150-instruction limit by moving general knowledge to separate `REFERENCE.md` files.
- A machine-readable `.rpi/journal.json` and human-readable `.rpi/SESSION.md` are automatically maintained by a `PostToolUse` hook.
- The `SessionStart` hook automatically re-seeds the context from `.rpi/journal.json` after a `/clear`.
- A standardized `Implementation Registry` is added to all implementation plan files to track phase progress (`[ ]`, `[/]`, `[x]`).
- New `/status` and `/resume` (aliased or renamed if conflict exists) commands provide session progress summaries and manual re-hydration triggers.
- No information loss occurs across at least three consecutive `/clear` and re-hydration cycles during a complex task.

## 🟡 Architecture

The system uses a "Hot/Cold" prompt strategy combined with "Journaled Checkpointing" to manage context pressure.

### Modular Skill Refactoring (Hot/Cold)
- **Hot Prompt:** `SKILL.md` files are limited to <150 instructions, focusing on active phase directives and mandatory constraints.
- **Cold Storage:** General programming knowledge and detailed house style rules (e.g., TypeScript conventions) are moved to `REFERENCE.md` files.
- **On-Demand Loading:** Agents are instructed to only `read_file` `REFERENCE.md` files when a specific task requires that domain knowledge.

### Automated Session Re-hydration
- **Journaling:** A `PostToolUse` hook monitors tool calls. At significant milestones (e.g., file writes, git commits, phase completions), it updates `.rpi/journal.json` (machine-readable) and `.rpi/SESSION.md` (human-readable).
- **SessionStart Re-hydration:** Every new session (including after `/clear`) begins with the `SessionStart` hook. This hook reads `.rpi/journal.json` and injects the essential state (Intent, Modified Files, Decisions, Current State, Next Steps) into the initial context.
- **Checkpointing:** When the `PostToolUse` hook detects "URGENT" context pressure (e.g., 70 tool calls), it forces a journal sync and provides the user with a command to `/clear` and continue.

### Implementation Registry
- Every implementation plan file (`docs/implementation-plans/*.md`) includes a standardized status table.
- Agents are required to update these markers (`[ ]`, `[/]`, `[x]`) as their primary task completion signal.
- The `PostToolUse` hook watches these files to update the global session status automatically.

## 🔵 Additional Considerations

- **Conflict with `/resume`:** Claude Code may already have a `/resume` command. If so, we will use `/rpi-resume` or `/continue`.
- **JSON Stability:** The `journal.json` structure must be simple and append-only where possible to avoid corruption.
- **Hook Latency:** The `PostToolUse` hook must be extremely fast to avoid adding lag to every tool interaction. We will use a lightweight shell script or compiled binary if needed.

## 🔵 Glossary
- **SKILL.md**: A file containing the primary instructions and constraints that define an agent's specific capability or "skill."
- **REFERENCE.md**: A supplemental file used for "Cold Storage" of general knowledge, house style rules, or detailed documentation, intended to be read only when required by the task.
- **Hot/Cold Prompt Strategy**: A context management pattern that keeps essential, frequently used instructions in the active prompt ("Hot") while moving auxiliary information to external files ("Cold").
- **Journaled Checkpointing**: The automated process of saving session state (intents, file changes, and decisions) to a local file (`.rpi/journal.json`) at significant milestones.
- **Re-hydration**: The process of automatically restoring a saved session state from a journal back into the agent's active context after a reset or `/clear`.
- **PostToolUse Hook**: An automated script triggered immediately after any tool execution, used here to monitor progress and update the session journal.
- **SessionStart Hook**: A script executed at the beginning of a new session or after a context wipe, responsible for initializing the environment and performing re-hydration.
- **Implementation Registry**: A standardized status table embedded in implementation plans to track the completion state (`[ ]`, `[/]`, `[x]`) of various project phases.
- **additionalContext**: A technical mechanism used by the agent's runtime to inject external data or previous session state into the initial conversation prompt.
- **/clear**: A CLI command that resets the agent's conversation history to free up context window space, which usually results in the loss of session state.
